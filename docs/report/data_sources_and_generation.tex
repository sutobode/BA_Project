\section{Data Sources and Data Generation}

This section covers the data foundation of the fraud-detection pipeline: the base dataset and its verified properties, the design and validation of thirteen synthetic contextual fields, and the resulting data dictionary.

\subsection{Base Dataset}
\label{subsec:base-dataset}

\paragraph{Source.} The base transactional data is the \emph{Online Payments Fraud Detection Dataset}\footnote{\url{https://www.kaggle.com/datasets/rupakroy/online-payments-fraud-detection-dataset}} (PaySim), a public Kaggle dataset produced by the agent-based mobile-money simulator of Lopez-Rojas, Elmir, and Axelsson (2016).\footnote{Lopez-Rojas, E. A., Elmir, A., \& Axelsson, S. (2016). PaySim: A financial mobile money simulator for fraud detection. In \emph{Proceedings of the 28th European Modeling and Simulation Symposium (EMSS)}.} The simulator models customers and merchants transacting over one month, with a subset of transactions labeled fraudulent to emulate account-takeover: an attacker gaining access to an account and transferring or withdrawing funds. This dataset was chosen because it has genuine transaction structure and a real class imbalance, unlike a synthetically balanced dataset that would understate the difficulty of the detection task.

\paragraph{Structure.} The dataset has \textbf{6,362,620 transaction records} across \textbf{11 columns} (Table~\ref{tab:paysim-columns}). No sampling was applied at any stage, to preserve the true class-imbalance ratio and avoid bias in the generation and validation steps described later.

\paragraph{Class imbalance.} Only \textbf{8,213 of 6,362,620 transactions (0.1291\%)} are labeled fraudulent --- the severe imbalance typical of real fraud problems. This ratio was measured directly from the raw file and used as a reference at every later stage, to confirm nothing in the pipeline silently altered the class distribution.

\paragraph{Empirical profiling.} Before designing any generation logic, the base dataset was profiled directly rather than assumed. Three checks were run against the full 6,362,620-row file: a cross-tabulation of the fraud label against transaction type; a cardinality check on \texttt{nameOrig} (unique values versus total rows); and a range/gap check on \texttt{step}. Three findings from this profiling determined the design in Section~\ref{subsec:synthetic-data}:

\begin{enumerate}
    \item \textbf{Fraud is confined to two transaction types.} Fraud occurs only in \texttt{TRANSFER} (4,097 of 532,909) and \texttt{CASH\_OUT} (4,116 of 2,237,500); \texttt{PAYMENT} (2,151,495), \texttt{CASH\_IN} (1,399,284), and \texttt{DEBIT} (41,432) contain zero fraud. This is account-takeover behavior --- unauthorized transfer or cash-out from a compromised account --- not point-of-sale or card-not-present fraud, and this distinction shaped every conditional field described later.
    \item \textbf{Customer transaction history is negligible.} \texttt{nameOrig} has 6,353,307 distinct values across 6,362,620 rows (99.85\% unique); of these, 6,344,009 rows (99.71\%) belong to an identifier appearing exactly once, and only 9,298 identifiers recur. A feature requiring a persistent, multi-transaction customer profile would therefore be built on almost no history. All synthetic fields were generated at the individual-transaction level instead.
    \item \textbf{The \texttt{step} field permits exact temporal derivation.} It encodes elapsed hours from a fixed start with no gaps across its 1--743 range, so time-of-day features can be computed deterministically rather than sampled --- removing an unnecessary source of randomness from the pipeline.
\end{enumerate}

This same profiling later fed a separately documented Data Cleaning stage, which confirmed the dataset is structurally clean and flagged (without removing) three fraud-relevant anomalies: zero-amount transactions, amount outliers, and a known balance-tracking characteristic of the simulator. That stage is reported separately.

\subsection{Synthetic Contextual Data}
\label{subsec:synthetic-data}

\paragraph{Motivation.} The base dataset has only raw financial fields --- it lacks the behavioral and e-commerce context a production fraud system would observe: device fingerprinting, IP-to-billing geolocation consistency, account tenure, payment-retry behavior, and temporal usage patterns. Thirteen contextual fields were generated in Python using \texttt{Faker} plus custom, documented business logic to close this gap.

\paragraph{Generation methodology.} All thirteen fields are fully vectorized \texttt{numpy}/\texttt{pandas} operations run on the complete dataset in one pass --- no row-wise iteration, no sub-sampling. All randomness comes from a single seeded generator (\texttt{numpy.random.Generator}, seed 42), so the pipeline is exactly reproducible; this was verified, not assumed. The code is organized into three modules (a geographic-distance utility, field generation, and leakage validation/reporting) and was developed test-first: sixty-four unit tests assert each field's statistical behavior, mathematical consistency, edge cases, and reproducibility; all sixty-four pass. Six design principles governed generation:

\begin{enumerate}
    \item \textbf{Row-level generation.} Given the negligible customer history (Section~\ref{subsec:base-dataset}), fields were generated per transaction, not through a persistent customer profile --- a departure from the entity-profile pattern used in commercial fraud systems, justified only because the data does not support it.
    \item \textbf{Label-free conditioning via an observable risk proxy.} No generation function reads \texttt{isFraud}. Fields with a plausible link to account-takeover fraud --- an unrecognized device, a geographically inconsistent IP, a young account, an atypical hour, repeated failed attempts --- are instead conditioned on a composite risk score computed only from observable transaction attributes available at scoring time: transaction type, the transaction's amount percentile within its own type, and whether it falls in the night window. This score deliberately excludes the account-balance columns (\texttt{oldbalanceOrg}, \texttt{newbalanceOrig}), since those columns near-perfectly determine \texttt{isFraud} in this simulator (fraud drains the account) and would otherwise leak the label through the back door. Fields without a behavioral rationale (browser, device type, device ID, billing country) were generated independently of the risk score. A dataset where every field correlates with the label would be unrealistic and would trivialize the classification task, so several fields were left deliberately uninformative. \texttt{isFraud} is used only after generation, to measure association, train models, and evaluate --- never as a generation input. This design was adopted after an earlier version conditioned directly on \texttt{isFraud}; that version could not be reproduced for a genuinely new transaction at scoring time (\texttt{isFraud} is exactly what is being predicted) and read as a textbook leakage pattern to anyone reviewing the code.
    \item \textbf{Bounded effect sizes.} Every conditional parameter was capped at a modest effect relative to baseline --- roughly 2--4\(\times\) for the Bernoulli and Poisson parameters, a partial shift for the one continuous parameter --- and documented as an explicit business assumption rather than calibrated against external ground truth, since none exists for this population. The bound exists specifically to stop a field becoming a near-deterministic proxy for the label, i.e.\ leakage.
    \item \textbf{Deterministic derivation where possible.} Fields computable from existing columns --- time-of-day, IP-to-billing distance, and the IP/billing country mismatch flag --- use a closed-form formula rather than random sampling, for accuracy and to guarantee internal consistency (the distance and mismatch fields are exactly zero/false whenever the two country fields match).
    \item \textbf{Post-hoc, objective leakage validation.} Every field's association with the fraud label was measured against a fixed numeric threshold after generation, rather than judged as merely ``looking reasonable.'' This check is retained even though generation is label-free, because a label-free proxy is not automatically leak-free: it must still be verified not to leak through the observable columns it does use.
\end{enumerate}

\paragraph{The risk proxy.} The risk score driving the five conditional fields above is a label-free composite in $[0,1]$: the equally-weighted average of (a) whether the transaction type is \texttt{TRANSFER} or \texttt{CASH\_OUT} --- the only channels this simulator's fraud uses, though the fraud rate within them is still under 1\%, making this a weak channel heuristic rather than a near-deterministic proxy; (b) the transaction's amount rank within its own type, a standard ``unusually large for this channel'' fraud heuristic; and (c) whether the transaction falls in the 00:00--05:59 window. All three components are ordinary observable transaction attributes, not derived from account balances or the label.

\paragraph{Leakage validation.} A feature that correlates too strongly with the fraud label would trivialize the classification task, so each field was validated quantitatively after generation. For continuous/Boolean fields, the univariate AUC against \texttt{isFraud} was computed as \(\max(\mathrm{AUC},\, 1-\mathrm{AUC})\) --- direction-agnostic, since a field that strongly predicts the \emph{absence} of fraud is just as concerning as one that predicts its presence. For categorical fields, Cram\'er's V was used, with Bergsma's (2013) small-sample bias correction\footnote{Bergsma, W. (2013). A bias-correction for Cram\'er's V and Tschuprow's T. \emph{Journal of the Korean Statistical Society}, 42(3), 323--328.} rather than the textbook formula. A field fails if its AUC reaches 0.75 or its corrected Cram\'er's V reaches 0.50 --- thresholds fixed before running the check, not adjusted afterward to force a pass.

\paragraph{Three issues found and fixed.} The validation procedure changed the generation logic three times; it was not a formality.

First, on the initial run against all 6,362,620 rows, \texttt{customer\_account\_age\_days} exceeded the AUC threshold (0.8753) at its original parameterization (fraud median 150 days versus baseline 400) --- too strong a separation to be a realistic signal. The fraud-conditional median was narrowed to 275 days (reducing, not removing, the effect) and the field regenerated; the AUC fell to 0.6689, below threshold.

Second, an independent review of the validation code found that the original (uncorrected) Cram\'er's V formula could report a spuriously high, sample-size-dependent value for the high-cardinality \texttt{device\_id} field (50,000 distinct values) --- as high as approximately 0.48 on a moderate-sized subset, close to the failure threshold, despite \texttt{device\_id} being generated completely independently of the label by construction. This was a contingency-table sparsity artifact, not a real relationship. The bias-corrected estimator was substituted everywhere Cram\'er's V is computed, with no change to the generation logic; \texttt{device\_id}'s corrected association on the full dataset is exactly 0.0000.

Third, a later technical review identified that five fields (\texttt{customer\_account\_age\_days}, \texttt{new\_device\_flag}, \texttt{ip\_country}, \texttt{shipping\_billing\_mismatch}, \texttt{failed\_payment\_attempts\_24h}) were, at that point, conditioned directly on \texttt{isFraud}. This was flagged for two reasons: such a field cannot be reconstructed for a genuinely new transaction at real-time scoring time, since \texttt{isFraud} is exactly what is being predicted and is not yet known; and reading the label directly inside a feature-generation function is a pattern any fraud-detection reviewer is trained to treat as a leakage smell, regardless of whether it is bounded. All five fields were rewritten to condition on the label-free risk proxy described above instead, and the dataset was regenerated from the original raw CSV. Table~\ref{tab:leakage-results} reports the results on the full dataset after all three fixes; all thirteen fields pass, with the five reconditioned fields now showing lower AUC (0.51--0.55, versus 0.54--0.67 previously) --- consistent with a weaker, non-circular, label-free signal, not a regression.

\paragraph{Effect on class distribution.} Generating the fields did not alter the class distribution: all 6,362,620 records and the 0.1291\% fraud rate are unchanged. Imbalance remediation (oversampling, class weighting) is deferred to feature engineering/modeling --- this stage only enriches the data. The enriched 24-column dataset was written out as input to the cleaning stage, along with a smaller stratified sample for quick inspection.

\paragraph{Limitations.} Two limitations apply specifically to the label-free redesign and should not be overclaimed. First, switching to a label-free risk proxy removes the leakage smell and the scoring-time reproducibility problem, but it does not make the data more realistic and does not remove circularity: the five conditional fields still carry fraud association by construction, via a hand-chosen proxy, not because they were learned from real behavioral data --- their measured association with \texttt{isFraud} in Table~\ref{tab:leakage-results} is evidence the injected correlation is bounded, not evidence these fields would be predictive on real-world fraud. Second, because the risk proxy is itself a function of \texttt{type}, \texttt{amount}, and \texttt{hour\_of\_day}, the five fields conditioned on it are statistically close to noisy functions of those three columns already present in the base dataset, and are unlikely to add much predictive value beyond them in downstream modeling; their purpose here is to illustrate the \emph{type} of signal a production fraud system would use, not to raise a classifier's AUC.

More generally, the conditional parameters in Table~\ref{tab:synthetic-fields} are explicit business assumptions, not values calibrated against external ground truth --- none exists for this population; this is an inherent limitation of synthetic augmentation on a dataset not originally collected with this context. Because the fraud label represents account-takeover confined to \texttt{TRANSFER}/\texttt{CASH\_OUT} (Section~\ref{subsec:base-dataset}), \texttt{shipping\_billing\_mismatch} was reframed from its literal e-commerce checkout meaning to an account-takeover one (registered address versus transaction address), to stay consistent with the dataset's actual fraud mechanism. All fields were generated at the transaction level (Design Principle 1); if this dataset ever gains genuine repeat-customer history, entity-level features (e.g.\ transaction velocity per account) could supplement the row-level fields here. Finally, a separate and more consequential risk sits outside this stage's scope: the base dataset's own \texttt{oldbalanceOrg}/\texttt{newbalanceOrig} columns near-determine \texttt{isFraud} in this simulator, so a near-1.0 AUC-PR reported in Model Development is more likely evidence of that pre-existing leakage than of strong modeling --- unrelated to the synthetic fields validated here, which measure 0.51--0.55 AUC and are confirmed non-leaky.

\subsection{Data Dictionary}
\label{subsec:data-dictionary}

\paragraph{Content and structure.} Every synthetic field is formally documented in a structured data dictionary covering all thirteen fields (Table~\ref{tab:synthetic-fields}). Each entry records eight attributes: column name, data type, unit, valid range, generation mechanism, the exact formula or distribution, the underlying business assumption, and the measured association with \texttt{isFraud} from Table~\ref{tab:leakage-results} --- labeled with its metric type (AUC or Cram\'er's V) so the two scales are not confused.

\paragraph{Automated generation.} The dictionary is generated programmatically by the same module that runs the leakage checks (Section~\ref{subsec:synthetic-data}), not transcribed by hand: it computes each field's association statistic and renders the dictionary directly from that result. This guarantees the documentation cannot drift from the code --- any parameter change is reflected automatically, and every association value is a live measurement, not a manually maintained estimate. This process caught one real defect during development: an early version reported association values without labeling which metric they were, which was fixed so every value is now explicit (e.g.\ ``0.6336 (AUC)'' versus ``0.0049 (Cram\'er's V)'').

\paragraph{Format and scope.} The dictionary is published in Markdown, satisfying the required deliverable format (Markdown or Excel), as a standalone artifact alongside the generation code. Table~\ref{tab:data-dict-excerpt} shows three representative entries; the full dictionary contains all thirteen fields.

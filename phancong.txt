Module
Focus Area
Key Tasks
1
Business Understanding & Data Generation
• Define the business problem and key KPIs (fraud rate,false-positive rate, financial loss avoided)
• Generate supplementary contextual/ behavioral data inPython
• Produce a complete data dictionary
2
Exploratory Data Analysis (EDA)
• Analyze distributions, outliers, and correlations
• Quantify class imbalance and compare fraud ratesacross transaction types, channels, and time windows
• Visualize fraud patterns (e.g., transaction amount vs.fraud likelihood, geographic mismatch patterns)
3
Data Cleaning
• Handle missing values, duplicates, and inconsistentcategories
• Treat outliers (e.g., implausible transaction amounts,invalid timestamps)
• Document all cleaning decisions with before/aftercomparisons
4
Feature Engineering
• Construct risk-signal features (velocity of transactionsper account, address mismatch flag, new-device flag,time-since-last-purchase)
• Encode categorical variables and scale numericalfeatures
• Address severe class imbalance (e.g., SMOTE,undersampling, class weights) and perform featureselection
5
Model Development
• Train and compare classifiers: Logistic Regression,Random Forest, XGBoost / LightGBM for fraudprediction
• Evaluate with metrics suited to imbalanced data: AUCPR, Precision/Recall, F1, and a cost-based metric (cost of missed fraud vs. cost of false alarms) 
• Select an operating threshold that reflects the business trade-off between fraud loss and customer friction
6
Model Deployment
• Package the model as an API (FastAPI or Flask) thatscores a transaction in real time
• Build a demo interface (Streamlit) simulating a fraudanalyst review queue
• Deploy to a cloud free tier (Render, Hugging FaceSpaces, or equivalent)
7
Model Monitoring
• Track data drift by comparing incoming transactiondistributions to training data
• Monitor performance metrics over time, includingprecision/recall on a rolling window
• Build a simple monitoring dashboard (e.g., using theopen-source Evidently library) and define retrainingtriggers
8
Final Report & Presentation
• Translate findings into risk-policy recommendations(e.g., which signals should trigger manual review vs.auto-block)
• Discuss limitations and future improvements
• Present to the class with a live or recorded demo
